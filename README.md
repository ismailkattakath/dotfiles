## Overview

A single source of truth for developer environment configuration that applies cleanly across
**local devcontainers, GitHub Codespaces, and the macOS host**. It manages git identity, SSH
defaults, editor consistency, the `gh` CLI, and host macOS terminal preferences — while keeping
secrets and personal data out of the repository by design.

Identity values (name, email, signing key) are injected at apply-time from local chezmoi data,
never hardcoded. A two-layer secret scanner (local pre-commit + CI) and a Claude Code guardrail
layer enforce the project's invariants automatically.

## Scope

| Surface | What it manages | Where it applies |
|---|---|---|
| **Dotfiles** | git identity & signing, SSH config, `editorconfig`, global gitignore, `gh` prefs, commit-message template | Linux, macOS, Codespaces |
| **macOS prefs** | Terminal.app & iTerm2 preferences (`macos/*.plist`) | macOS host only (no-op on Linux) |

## Quick start

```bash
# One-line bootstrap (installs chezmoi if missing, then applies)
sh -c "$(curl -fsLS get.chezmoi.io)" -- init --apply ismailkattakath

# Or, if you already have the repo cloned to ~/dotfiles:
chezmoi init --source ~/dotfiles
chezmoi diff      # preview
chezmoi apply     # apply source → $HOME
```

In **devcontainers / Codespaces**, `install.sh` is invoked by the dotfiles feature; it installs
chezmoi, resolves the identity vars the templates reference (`name`, `email`, `signingKey`), then
applies. Identity is **never hardcoded to a person** — it is resolved in order:

1. **Explicit env vars** — `GIT_AUTHOR_NAME` / `GIT_AUTHOR_EMAIL` / `GIT_SIGNING_KEY`
2. **The GitHub account** — auto-derived via `gh` (Codespaces authenticates it with `GITHUB_TOKEN`):
   `name` from the account, `email` as the account's `<id>+<login>@users.noreply.github.com`
3. **Existing git config** — whatever `git config --global` already knows

Signing: in **Codespaces** the script defers to GitHub's own commit-signing setup (the
[GPG-verification docs](https://docs.github.com/en/codespaces/managing-your-codespaces/managing-gpg-verification-for-github-codespaces)
warn that dotfiles git config "may conflict with the configuration that GitHub Codespaces requires
to sign commits"), so it drops our SSH-signing override and lets GitHub win. In other non-interactive
envs (local devcontainers / CI) SSH signing stays on only when a `GIT_SIGNING_KEY` is actually
present — otherwise it is disabled so `git commit` works out of the box.

## How identity injection works

Templates never contain real identity values — they reference chezmoi data:

```gitconfig
[user]
    name       = {{ .name }}
    email      = {{ .email }}
    signingkey = {{ .signingKey }}
```

Values are defined per-environment:

- **Host:** `~/.config/chezmoi/chezmoi.toml` `[data]` (git-ignored, never committed)
- **Containers:** generated by `install.sh` from `GIT_AUTHOR_NAME` / `GIT_AUTHOR_EMAIL` / `GIT_SIGNING_KEY`

## Security

This repo treats configuration as a potential leak surface and guards it on three fronts:

- **No secrets, ever** — tokens live in `~/.secrets` (host) or Codespaces secrets; never tracked.
- **No hardcoded personal data** — emails come from chezmoi data vars or GitHub noreply; macOS
  plists are scrubbed of usernames and security-scoped path bookmarks before commit.
- **Two-layer scanning** — [gitleaks](https://github.com/gitleaks/gitleaks) runs both as a
  `.githooks/pre-commit` hook and as a CI workflow on every push. CI runs on a **self-hosted
  runner** (owned hardware), not GitHub-hosted compute.
- **Signed commits** — SSH commit signing (`gpg.format = ssh`); the credential helper uses
  `!gh auth git-credential` so it resolves `gh` from `PATH` on any platform.

Run `/audit` (Claude Code) before pushing to scan for secrets, emails, and PII.

## Claude Code integration

The `.claude/` directory makes this repo self-documenting and self-guarding when edited with
[Claude Code](https://claude.com/claude-code):

| Command | Purpose |
|---|---|
| `/status` | `chezmoi status` + diff — what differs between source and `$HOME` |
| `/apply` | Apply source → home, then commit & push source changes |
| `/add <path>` | Track a new dotfile (after invariant checks) |
| `/audit` | Scan for secrets, emails, tokens, and PII before pushing |

**Guardrails** (`.claude/settings.json` hooks) block, automatically: `{{ if }}` platform-switching
templates, hardcoded secrets/emails, shell-config files, and destructive git commands
(`rm -rf`, `git reset --hard`, `--no-verify`). Design rules: `.claude/rules/design-invariants.md`.

## Conventions

- File prefix mapping: `dot_X` → `~/.X`, `dot_config/Y` → `~/.config/Y`
- Templates (`*.tmpl`) only for identity vars or the macOS-prefs change-guard — **never** `{{ if }}` blocks
- No `.zshrc` / `.zprofile` / `Brewfile` — shell setup is owned by devcontainer features
- Machine-specific SSH hosts go in `~/.ssh/config.local` (git-ignored), included by `dot_ssh/config`
